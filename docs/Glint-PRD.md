## 1. 产品概述

### 1.1 一句话定位

Glint 是一个面向**非纯技术背景使用者**（产品经理 / 产品角色）的代码理解与学习工具：它不教你写代码，而是帮你**读懂代码背后的工程决策与设计意图**。它消灭"选中→复制→拖给 AI 解释"这种又慢又傻的操作——**你当前选中/光标所在的对象，本身就是给 AI 的输入；按一个 Option 数字键，对应维度的理解即刻就位**。

### 1.2 背景与问题陈述

产品角色每天与工程师协作、读技术方案、评审实现，却常卡在三处：**读不懂语法**（不知道每行在做什么、为什么这么写）、**看不清组织**（说不清为什么这样拆模块、对业务意味着什么）、**建立不起框架感**（不清楚用了什么语言/技术栈/框架及其关系）。

现有工具填不上这个空白。IDE（如 VS Code）擅长浏览编辑，但默认你已经懂代码；通用 AI 助手能解释片段，但要反复手动粘贴、缺项目级上下文、答案常常太笼统。更根本的是——**它只能解释「点」，不能帮你建立「网」**。研究中一位审计员一句话戳破：「我可以让模型解释一个函数，但它从不告诉我这个函数怎么跟系统的其他部分连起来。」Glint 要做的，是把"理解代码"的操作成本压到近乎为零，并让理解能在代码/调用/执行/架构四个角度间瞬间切换、还能留下"我看过哪里"的痕迹。

### 1.3 目标用户与画像

**主要用户 · 产品角色（本阶段即使用者本人）。** 有产品/业务背景，能看懂技术讨论的「是什么」，但缺乏系统编程训练，难以独立拆解「为什么这么实现」。诉求是在协作、读方案、做决策时，快速自助把一段代码或一个项目读明白，并逐步积累工程直觉。研究表明，这类**新手的核心障碍不是看不懂答案，而是不知道该问什么、容易迷失、不知道自己看过哪/漏看了什么**——这直接决定 Glint 必须主动给"起点"、给"该看的维度"、并把理解路径留痕。

**次要用户 · 上线后的同类人群。** 其他想读懂代码的非工程背景者。该群体在上线阶段才正式服务（带来 GitHub 导入、多用户、前端配置 key 等需求，本阶段不做）。

### 1.4 核心价值主张

与「代码解释器」类工具相比，Glint 的差异化有四点。其一，**零搬运 · 选中即懂**：当前选中即输入，按 Option 数字键即得理解，不用复制、不用打字、不用拖拽（§4）。其二，**同一焦点、四个角度瞬间切换**：⌥1 这段代码为什么这么写 / ⌥2 它怎么被调用 / ⌥3 一件事依次怎么执行 / ⌥4 它在整体架构的哪个位置，焦点不变、只换角度看（先给网再给点，对应论文"跨抽象层无缝切换不丢上下文"）。其三，**理解留痕**：看过的地方收纳成一条"理解轨迹"，随时回看，专治新手"迷失/不知看过哪"。其四，**主动降低提问门槛**：选中即懂 + 每层预置关键信息，新手无需先学会提问即可获得理解。配合记录、概念标签与交互轨迹，Glint 进一步从「代码解释器」演化为「个人化的代码学习教练」（见 §8）。

## 2. 研究依据（为什么这样设计）

Glint 的核心设计借鉴论文 *Understanding Codebase like a Professional!*（Gao et al., arXiv:2504.04553v2）。研究者研究的是**代码审计员**——一种每天都要快速读懂全新代码库的工种，沉淀了别人没有的"快速读懂代码"方法论。研究访谈 8 位审计员，提炼方法并实现原型 **CodeMap** 验证。对 Glint 最关键的发现：

**① 理解是层级化、可往返的。** 审计员遵循「全局→局部→细节」的理解流，并在各层间反复跳转、不断修正心智模型。对应 Glint 的架构/模块/代码三层，以及"同一焦点四维切换"。

**② 普通 AI 的三个痛点。** 反复手动 prompt 才能拿到分层信息；要自己把大项目拆成小段喂 AI；答案常不相关/太笼统。根因是它能定位「点」却无法表达结构「关系」。

**③ 四个设计机会。** 信息抽取贴合理解流；信息分解贴合心智模型；信息表示跨层连接（表达跨模块依赖）；以及降低提问门槛——为新手自动展示预定义信息与常用细节，无需自己组织 query。

**④ 实验结论。** 与 ChatGPT 比，用 CodeMap 的人花在"读 AI 文字回复"上的时间下降 **79%**，转而更多靠看图建立理解；与传统静态可视化工具比，看系统图的时间多 **90%**。由 9 名熟练 + 6 名新手验证；对新手显著提升理解、减少漫无目的乱看。

**⑤ 准确性工程做法。** AI 直接生成结构图不可靠（漏文件、幻觉）。CodeMap 用**迭代精修 + 用「文件名/数量」这种固定事实当 ground truth 校验**（图里没有错误文件即视为准确），约 3–4 轮收敛。Glint 复用这套"凡可数必跑码、AI 自判不算数"思路（见 Spec）。

Glint 映射：选中+Option 数字键（§4）把"建立网、跨层、降门槛"压进零搬运手势；⌥2/⌥3/⌥4 三类图回应①③⑤；选中即降噪（§4.8）回应②；预置信息 + 选中即懂回应③④；理解轨迹（§4.5）回应"新手迷失"痛点。

## 3. 核心理念：三层教学 + 可往返

Glint 从三层理解任意代码，三层既是讲解维度，也是数据与交互组织维度，与论文"全局/局部/细节"对应，并强调**层间可往返、心智模型可迭代修正**。四个 Option 维度即是这三层的可操作化（⌥1 偏代码层，⌥2/⌥3 偏模块层，⌥4 偏架构层）。

| Glint 层 | 对应论文层 | 讲什么 | 主要承载维度 |
| --- | --- | --- | --- |
| 架构层 | 全局 | 整体架构选型、技术栈/框架及关系 | ⌥4 架构鸟瞰 |
| 模块层 | 局部 | 模块如何组织、如何参与业务、为什么这样组织 | ⌥2 调用关系 / ⌥3 执行路径 |
| 代码层 | 细节 | 逐行逻辑、为什么这么写、数据流 | ⌥1 为什么这么写 |

## 4. 核心交互：选中 + Option 数字键（已拍板，不得改设计意图）

### 4.1 一句话与核心规则

消灭"选中→复制→拖给 AI"。**使用者当前选中/光标焦点本身就是给 AI 的输入**，按一个键，对应维度的理解即刻就位，零搬运。核心规则：**这套快捷键永远作用在"当前选中的对象"上**，无需再指明要解释谁——选中一行代码就解释这行，选中一个模块节点就展开这个模块。

### 4.2 四个维度快捷键（语境感知）

| 快捷键 | 看什么 | 呈现形态（见 §5.2） |
| --- | --- | --- |
| **Option + 1** | 这段/这行代码**为什么这么写**：语言、语法、逻辑、它在文件/函数里的位置与作用 | **就近浮卡**（贴着被选中代码弹出，轻量） |
| **Option + 2** | 这个模块/函数**怎么被调用**：谁调用它、它调用谁 | **独立面板 · 调用关系图**（节点+连线，连线配一句话解释） |
| **Option + 3** | 这个模块的**执行路径**：一件事依次走了哪几步 | **独立面板 · 流程/时序图** |
| **Option + 4** | 整个模块/项目的**架构**：全景、分块、技术栈及关系 | **独立面板 · 架构鸟瞰图**（可缩放区块版图 / Treemap） |

### 4.3 四维瞬间来回切，永不丢焦点

同一个选中对象，四个键自由往返：看着一段代码 `⌥1` 懂语法 → 立刻 `⌥2` 看它所在模块怎么被调用 → `⌥4` 拉远看它在整个架构的哪个位置。**焦点对象始终不变，只是换角度看它。** 这对应论文核心发现：在抽象层之间无缝切换、不丢上下文。

### 4.4 ⌥1「为什么这么写」是最高诉求（不可弱化）

使用者反复强调：**必须能明白每一行代码为什么那样写。** ⌥1 是本产品对使用者个人最关键的功能。其卡片至少回答：这是什么语言、用了什么语法、逻辑怎么走、这行在它的上下文（函数/文件/模块）里处于什么位置、起什么作用。可附"同类写法在别处哪里也出现"（泛化，P1 可后置）。

### 4.5 卡片收纳成"理解轨迹"（使用者设计，取代旧版 Space 钉住）

- Option 唤出的卡片**固定停在原地**——不是悬停飘过即走，因为小白需要时间慢慢读，轻量预览对他无用。
- 当使用者在**新位置**再次按 Option 时，**旧卡片不消失，而是收缩成一个小标签**（极简显示，像收起的标签页），新位置弹出新卡片。
- 这些收起的小标签**串成一条"理解轨迹"**——Option 扫过的路径留下一排标签，标记"我理解过这里、这里、这里"，专治新手"迷失、不知看过哪、漏看了什么"。
- 点击任一收起标签 = **重新展开成完整卡片**（可召回的历史）。

### 4.6 钻取 = 在更细对象上再 Option 一次（不引入新键）

- **不设 Option+Space 两个键**——过度设计，破坏"只记一个动作"的原则。
- "往下钻"不设专门键：想看某模块内部，就把焦点移到更细的对象（模块里的某个文件/函数）上再 Option 一次；新卡片弹出、旧卡片自动收纳成标签。**钻取 = 在更细粒度对象上再触发一次同样的手势。**
- 补充（非必须）：浮卡里的关系词（如"被 api 调用""依赖 session"）可做成可点链接，点击即跳到该对象，作为钻取的便捷补充，但不是必需键。

### 4.7 卡片内容随焦点粒度自适应

同一手势，焦点粒度不同，给的那一层答案不同。来源分层以规避成本（见 §4.8）：

| 焦点对象 | 卡片告诉什么 | 内容来源 |
| --- | --- | --- |
| 文件夹/模块 | 在业务里负责什么、跟谁连、是否入口 | 预生成，库内直读（瞬时、≈0 成本） |
| 文件名 | 在所属模块扮演什么、被谁调用、调用谁 | 预生成，库内直读 |
| 函数/类 | 做什么、谁触发、输出去哪 | 首次触发懒生成并缓存 |
| 选中的代码（行/段） | 这几行干什么、为什么这么写 | 实时调 AI |
| 变量名 | 从哪来、谁读/谁改 | AST 静态分析（确定、免费、瞬时） |

### 4.8 成本红线（必须遵守）

绝不能每次 Option 都实时调 AI（又慢又烧 token，丝滑感全失）。**模块层/文件层的角色说明在预生成阶段就备好存库，触发时直读库、瞬时呈现**；变量引用走 AST；函数/类首次触发才生成并缓存；**只有"选中代码的逐行解释"和"向 chatbot 追问"才实时调 AI**。

## 5. 整体布局：VS Code 式可停靠多面板（已拍板）

### 5.1 布局要求

使用者已认可 VS Code 那种信息密集的工作方式，因此采用**多视图并排的 IDE 式布局**（不是"单画布切换"的极简地图式）。具体：① **文件树必有**、**具体文件的代码预览必有**；② 各视图（调用关系图、执行路径图、架构图、代码、卡片面板等）**都可收起、可来回切换**；③ 整体像 VS Code——可开多个面板/边栏，**且能方便地拖动改变位置**（可停靠、可分屏、可标签化）。

### 5.2 呈现的轻重分法（已拍板）

- **⌥1（语法/为什么这么写）= 就近浮卡。** 轻量，贴着代码弹出，看完即走或收纳成轨迹标签。**不做成占整块面板的视图。**
- **⌥2 / ⌥3 / ⌥4（调用关系 / 执行路径 / 架构）= 各自独立拉出来的面板视图。** 这类图天然需要大空间铺开，必须单独成面板。（使用者原话：调用关系"就得单独拉出来"，语法解释"可以就近弄"。）

### 5.3 三种图是三种不同性质的结构（别用一张图硬塞所有东西）

- **架构/包含关系（⌥4）= "地图/版图"性质** → 嵌套区块图（Treemap），回答"项目长什么样、分几块、各部分多大"。
- **调用/依赖关系（⌥2）= "网络/线路"性质** → 节点+连线有向图，回答"谁调谁、一个动作牵动哪些部分"；每条连线**配一句自然语言解释**（图管结构、话管意义）。
- **执行路径/时序（⌥3）= "流程/故事线"性质** → 流程/泳道时序图，回答"一件事依次怎么发生"，对应"模块如何参与业务逻辑"的诉求。

### 5.4 视觉调性

文件树与所有图视图均**视觉自有设计、不照搬 VS Code 长相**（使用者认为 VS Code 太丑）。调性见《design-system/Glint-Design-System.md》：**电蓝 + 黑灰、近乎单色只留一个强调色、Pascal/终端字符的像素点缀，结构与组件向 Linear 看齐**。设计稿/原型由使用者本人产出，最终以使用者设计为准；本阶段下一个 agent 不需敲定最终视觉，应聚焦信息架构、组件结构（可换肤封装）与接口预留。

### 5.5 Edge bar 与侧栏（导航骨架）

最左是一条竖向图标栏 **Edge bar**（交互参考 VS Code 的 Activity Bar、视觉自有设计），承载全局/项目级入口；点击图标在其右侧展开/收起对应**侧栏面板**。建议入口：**项目切换、文件树、搜索、技术栈、Agent、理解轨迹、设置**。侧栏可整体收起以让出空间。整体骨架 ＝ `Edge bar（竖向图标栏）` ＋ `侧栏（可收起）` ＋ `中部可停靠面板区（代码 / ⌥2 / ⌥3 / ⌥4）` ＋ `⌥1 浮层（不占面板）`。

### 5.6 技术栈认知（Tech Stack · Edge bar 面板 + ⌥4 联动）

回应"先搞清这个仓库用了什么语言/框架，再逐个弄懂它们"的诉求，是架构层"框架感"的落地。

**入口与列表。** Edge bar 的「技术栈」面板自动列出本仓库检测到的**语言 / 框架 / 库**（如 TypeScript、React、jQuery、JSON、PostgreSQL…），按类型分组。

**每项点开 ＝ 一张认知卡片**，答四件事：① **是什么**——一句话定义；② **用途**——解决什么问题、典型场景；③ **在整个代码体系中的位置**——属于哪一层、和谁同代/替代/上下游（例：jQuery 是老牌前端 DOM 操作库，位于原生 JS 之上、现代框架 React/Vue 之前的一代）；④ **在本仓库的角色与关键位置**——这里哪儿用到它、扮演什么关键角色，并给**跳到使用处**的链接。

**与 ⌥4 联动。** 卡片里"角色/使用处"与架构鸟瞰（⌥4）打通：技术项 ↔ 架构里的位置互跳；顺"跳到使用处"再用 ⌥2/⌥3 即可看它与其他部分的具体联系——正好串起"了解它与其他部分的联系、在框架里的关键位置"。

**先文字后图。** ①②③ 是通用知识，先用文字卡片表达；"技术全景/分层图"可视化后置（见 §10）。

**成本（承接 §4.8）。** ①②③ 是**项目无关的通用知识**——"jQuery 是什么"对所有项目都一样，**跨项目缓存一次、近乎免费**；④ 来自预生成的架构分析。检测走清单/扩展名（`package.json`、`requirements.txt`、`go.mod`、文件后缀…）**确定性识别**，不靠模型猜。

### 5.7 Agent Bar（开放式探索 · P0）

四维 Option 键解决"对当前焦点的**结构化**理解"；**Agent Bar 解决"跨焦点、开放式的探索"**——那些不落在单个对象/单个维度上的问题（如"这个项目怎么做鉴权的？""支付流程牵动哪些模块？""我改这个函数会影响什么？"）。两者互补：Option 键快而结构化，Agent Bar 广而开放。

**形态。** 像 VS Code 的面板——**可停靠、可展开收起、随拉随走**；从 Edge bar 的「Agent」入口或快捷键（如 ⌘K / ⌘L）唤起，输入自然语言即探索；结果在面板内流式呈现，可常驻保留多轮上下文，也可收起让位。

**能力：读 + 驱动界面 + 主动建议。**
- **读**：检索代码、跟踪调用/依赖、综合回答，每条结论附**可跳转引用**（文件 / 符号 / 图节点）。
- **驱动界面**：能替你**打开面板、跳转焦点、在图上高亮、触发某个 ⌥ 维度**——把"该看什么"直接呈现到界面上，而非只给文字。
- **主动建议**：回答后主动给"**接下来看哪 / 你可能漏了什么**"的探索路径建议（呼应论文"新手不知道该问什么、容易迷失"）。

**与其它部分的关系。** Agent Bar 的检索底座 ＝ 全量 RAG（F24）＋ 既有符号/调用/结构数据 ＋ 四维服务（它可像调用工具一样触发 ⌥1–⌥4）。它产生的探索路径同样进入交互轨迹与成长分析。

**边界。** 默认**只读 + 驱动界面**（导航类动作即时执行）；任何写/改/外部副作用类动作必须二次确认（本阶段基本无写操作）。自主程度上限见 §10。

## 7. 关键用户流程

**流程一 · 导入即见框架。** 上传本地文件夹 → 系统索引、符号解析、识别语言与技术栈 → 后台预生成（架构+模块+各元素角色卡片）并迭代生成结构（含文件名/数量校验）→ 完成后进入 IDE 式布局：文件树 + 代码预览 + 默认的架构鸟瞰（⌥4），新手照入口与阅读指引看起。

**流程二 · 选中 + Option 四维理解。** 在代码/文件树/图节点上**选中**某对象 → 按 `⌥1` 看"为什么这么写"（就近浮卡）→ 同一焦点按 `⌥2` 拉出调用关系面板 → 按 `⌥4` 拉远看架构位置 → 按 `⌥3` 看它参与的执行路径。焦点不变、四角度自由切。想深入：把焦点移到更细对象（某文件/函数）再 Option 一次。

**流程三 · 留痕与回看。** 每次在新位置 Option，旧卡片收纳成轨迹标签，排成"我理解过的路径"；点任一标签即重新展开。全部解释、追问与交互轨迹入库并打标签，积累成长分析信号。

**流程四 · 技术栈认知。** 从 Edge bar「技术栈」面板看仓库用了哪些语言/框架/库 → 点开任一项（如 jQuery）读认知卡片（是什么 / 用途 / 生态位置 / 在本仓库角色）→ 顺"跳到使用处"用 ⌥2/⌥4 看它与其他部分的联系与关键位置。

**流程五 · Agent Bar 开放探索。** 唤起 Agent Bar → 问"这个项目怎么做鉴权的？"→ Agent 检索相关代码与调用链、综合回答并附跳转引用，同时**自动打开 ⌥2 调用图并高亮鉴权相关节点** → 末尾建议"接下来可看 session 模块 / 你可能漏了中间件 X" → 顺着点进去继续用四维细看。探索路径进入轨迹。

## 8. 成长分析（产品愿景）

通过分析两类信号——**解释记录的概念标签** + **交互轨迹**（在哪些对象反复 Option、停在哪个维度、轨迹怎么走、哪些地方反复回看）——识别使用者经常不懂、反复纠结的知识弱项，给针对性补强。交互轨迹比"问了什么"更结构化、更干净，是关键优势。前提是数据准备：记录必须打概念/知识点标签，交互事件需带对象粒度、维度与层级。Spec 已预留相关表并要求**从 MVP 第一天就采集**（分析功能 P1 才做）。典型呈现（P1/P2）：弱项看板、针对弱项的补强建议、掌握度随时间变化。

## 9. 数据与隐私

后端持久化至少含：用户数据（账户、项目）、解释记录（含概念标签）、交互轨迹/事件。隐私上，本阶段个人单用户、Key 留后端、数据存使用者自己的后端。上传代码可能含敏感信息，仅用于本工具解释与检索；调用第三方模型时片段会发往对应服务商，设置中可见并可选服务商；上线多用户阶段补数据隔离、删项目即删其代码与记录。
