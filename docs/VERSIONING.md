> 对所有产品/技术文档统一做版本管理，像 Git 一样**可追溯、有清晰的版本标识与历史**。

---

## 1. 版本标识

格式：`vMAJOR.MINOR` ＋ 日期 `YYYY-MM-DD`，完整写法如 **`v0.7 · 2026-06-20`**。

- `MINOR + 1`：每次实质性修订（新增/改动一块内容）。
- `MAJOR + 1`：大结构性改版（如核心交互被推翻重做）。
- **同一天多个版本**：靠版本号区分（如同日 v0.6 → v0.7），日期相同不冲突——与 Git 同一天多次提交同理。
- **各文档版本号独立演进**：PRD 与 Spec 不必同步，因为变更频率不同。这正是 Git 式"按文件各自追踪历史"。

## 2. 每份文档必备三处

1. **头部表**：`版本`、`最后更新`（日期）。
2. **底部「修订历史」表**：`版本 | 日期 | 摘要`——该文档自己的版本时间线（最新在上）。
3. **顶部变更摘要**（紧跟头部）：记最近一两版要点。

## 3. 跨文档时间线

`CHANGELOG.md` 记录所有文档的**统一时间线**（按版本/日期倒序），一眼看清"这一批动了哪些文档"。与各文档的「修订历史」互补：前者跨文档，后者单文档。

## 4. Git

- 每次修订 ＝ 一个 commit（Conventional Commits：`docs:` / `feat(docs):` / `fix(docs):`）。
- 里程碑打 **annotated tag**：`docs-YYYY.MM.DD`（同日多个加 `-2`、`-3`…），tag message 记录当时各文档版本。
- 文档与代码同仓，文档版本管理 ＝ 仓库 git 历史的一部分。

## 5. 工作流（每次改文档照此走）

1. 改内容。
2. 升该文档版本号 ＋ 更新「最后更新」日期 ＋ 更新顶部变更摘要。
3. 在该文档「修订历史」加一行。
4. 更新 `CHANGELOG.md`。
5. `git commit`（里程碑再 `git tag`）。
